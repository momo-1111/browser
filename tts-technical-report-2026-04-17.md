# Browser 项目 TTS 实现与体验问题技术文档

## 1. 文档目的
本文件用于汇总当前 TTS 功能的实际实现方式、运行链路、诊断能力、已知体验问题与后续优化建议，供开发、测试和产品对齐。

## 2. 范围与版本
- 项目：OpenHarmony ArkTS Browser
- 核心实现文件：
  - entry/src/main/ets/pages/Index.ets
  - entry/src/main/ets/view/SettingDialog.ets
  - entry/src/main/ets/utils/ReaderExtractionService.ets
- 统计时间：2026-04-17

## 3. 当前实现总览
### 3.1 启动入口
- 设置弹窗中“朗读页面 -> 开始”通过递增 ttsRequestToken 触发主页面监听。
- 主页面 onTtsRequest 收到信号后进入 startTtsFromCurrentPage。

### 3.2 文本来源
- Reader 模式：直接使用 readerParagraphs 构建句子单元。
- 非 Reader 模式：通过页面可见文本抽取（readerTextScript）构建句子单元。
- 句子处理策略：清洗 HTML 噪声、按中文标点切句、长句按逗号再拆、去重与低质量文本过滤。

### 3.3 发声链路（三级）
- 一级：Web SpeechSynthesis（本机 voices > 0）
- 二级：Web Audio 云端音频兜底（voices = 0）
- 三级：原生 AVPlayer 兜底（Web 音频触发失败时）

### 3.3.1 引擎策略（新增）
- 当前固定默认策略：原生优先（AVPlayer 主线路）。
- 当原生链路失败时，自动回退到 Web 语音链路，保证会话不中断。

### 3.4 播放控制
- 核心控制：播放、暂停、上一句、下一句、停止。
- 连播策略：
  - native 模式按播放器完成状态推进下一句，避免前后句重叠。
  - speech 模式按估算时长切句。
  - audio 模式按 onended 事件推进下一句，避免“按时长提前切句”造成打断。

### 3.5 并发与竞态防护
- ttsPlayGeneration：会话代次锁，丢弃旧异步回调。
- ttsCommandLocked：按钮操作锁，防止短时间重复点击导致并发。
- __readerAudioActivePlay + __readerUnlocking：区分“真实播放”与“解锁播放”，避免解锁阶段误触发下一句。

### 3.6 日志与诊断
- UI 诊断项：状态文本、调试串、最近操作日志、最近日志明细。
- 文件日志：写入应用沙箱 cacheDir 下 tts_runtime.log。
- 关键日志关键词：
  - start
  - resume-audio
  - auto-next
  - probe
  - tap-resume
  - native-ok / native-fail

## 4. 关键方法说明（Index.ets）
- startTtsFromCurrentPage：初始化会话、构建朗读单元、准备发声环境。
- buildHighlightOnlyScript：仅执行高亮，不触发网页发声，供原生主通道使用。
- buildSpeakScript：在 WebView 内构建 speech 或 audio 通道，并注入高亮。
- triggerAudioPlayback：执行音频解锁、播放触发、延迟探针判定，失败时挂载页面点击恢复监听。
- armTapResumeScript：播放被拦截时，注入一次性页面手势监听；用户点击网页任意位置后自动恢复播放。
- tryNativeFallback：Web 音频失败时尝试原生 AVPlayer 兜底，并回写状态机。
- scheduleNextTts：根据当前模式驱动自动连播。
- pauseTts / resumeTts / stopTts：统一的控制态切换与资源清理。

## 5. 当前“影响实际操作体验”的核心问题
### 问题现象
- 用户在控制条点击“播放”后，仍可能无声。
- 用户点击网页空白区域后，音频恢复正常。
- 个别场景会出现首句触发不稳定或需要额外交互。

### 问题本质
- WebView 自动播放策略要求页面侧用户激活（user activation）。
- ArkUI 控件点击不总能等价传递为网页侧激活事件。
- 因此在 ua=0 时，即使执行 audio.play，也可能被内核拦截。

### 现阶段已做缓解
- 加入页面点击自动恢复（armTapResumeScript）：只要点击网页一次即可自动续播，无需再次点播放。
- 增加原生兜底链路：Web 音频无法触发时尝试 AVPlayer。
- 增加探针与日志明细：可定位 blocked、retry、audio_play、audio_end 等状态。
- 已实现“原生优先 + 自动回退”策略：默认优先原生播放，失败自动回退 Web。

### 仍未完全消除的体验风险
- 由于策略属于 WebView 内核安全机制，无法保证“零额外交互”在所有站点都成立。
- 原生兜底在当前环境仍存在失败样本（native-fail），成功率受网络源、格式、设备能力影响。

## 6. 现状评估
- 功能完整性：已具备可用的句级朗读、控制、连播、日志诊断。
- 稳定性：较早期显著提升，主要竞态（解锁误推进下一句）已做防护。
- 体验短板：原生链路在个别设备/音源组合下仍可能失败，回退到 Web 后仍可能受网页手势策略影响。

## 7. 后续优化建议（按优先级）
### P0（必须）
- 建立“页面激活后自动恢复”的强状态回路：
  - 被拦截后持续短时监听页面手势，触发即续播并自动销毁监听。
  - 若连续 N 次 blocked，直接切换清晰提示和兜底策略。

### 7.1 问题根源总结
- 当前“点击播放后无声、点击网页后有声”的根因不在业务逻辑，而在 WebView 内核安全策略。
- Web Speech/Web Audio 在 WebView 中受 User Activation 约束，要求网页内部手势激活。
- ArkUI 按钮点击不总是等价为网页内部有效激活，因此可能出现 `audio.play()` 被拦截。
- 结论：只要播放核心继续依赖 WebView 内部发声，就会持续存在概率性无声与交互不一致问题。

### 7.2 最优目标方案（架构级）
- 将“播放核心”迁移到 ArkTS 原生 TTS（`@ohos.tts`）主线路。
- Web 侧保留文本抽取与高亮，不再承担发声职责。
- 目标链路：
  - 文本抽取（Web/Reader）
  - 句子列表传入 ArkTS
  - ArkTS 原生 TTS 播放（主线路）
  - 进度回调驱动 Web 高亮

### 7.3 方案优势与边界
- 优势：
  - ArkUI 点击即播放，不依赖网页手势。
  - 从根源规避 WebView 自动播放拦截。
  - 播放状态由 ArkTS 统一管理，状态机更清晰。
  - 离线场景能力更可控（依赖系统 TTS 语音资源）。
- 边界：
  - 需确认当前 SDK/设备是否提供可用的 `@ohos.tts` 能力与权限要求。
  - 若运行环境缺失原生 TTS 模块，必须保留现有 Web 链路作为回退。

### 7.4 落地方法（可执行）
- 步骤 1：复用现有文本抽取与分句逻辑，输出 `ttsUnits`。
- 步骤 2：新增 NativeTtsService 封装（初始化、speak、pause、resume、stop、onEnd/onBoundary）。
- 步骤 3：在 `Index.ets` 中采用固定默认策略：
  - 主线路：Native TTS（当前实现为 AVPlayer）
  - 回退线路：当前 Web Speech/Web Audio（自动触发）
- 步骤 4：高亮改为“ArkTS 播放进度驱动 Web 高亮”，而不是“Web 发声回调驱动高亮”。
- 步骤 5：保留自动回退链路与最小可观测日志，确保线上可快速定位问题。

### P1（高优先）
- 强化原生主通道可行性验证：
  - 评估是否将原生播放提升为默认通道，Web 仅做文本高亮与控制。
  - 统一音源与格式，降低 native-fail。
  - 条件允许时评估接入 @ohos.tts 作为系统级主通道（当前实现基于 AVPlayer）。

### P2（增强）
- 增加“导出日志文本”按钮（复制到剪贴板或保存到用户可访问目录）。
- 对日志字段做结构化（JSON 行格式），便于自动分析回归。

## 8. 测试与验收建议
- 场景 A：首次点击播放即出声（目标）
- 场景 B：首次 blocked 后，点击网页一次自动恢复（当前最关键）
- 场景 C：上一句/下一句在 audio 模式下可稳定切换，无重叠声
- 场景 D：暂停/恢复不丢句、不重复句
- 场景 E：停止后重开，状态和日志可重置

## 9. 结论
当前 TTS 方案已经形成“Web speech -> Web audio -> Native”多级回退，并具备可观测与可诊断能力。当前最影响体验的问题不是业务逻辑缺失，而是 WebView 自动播放策略导致的用户激活约束。现版本通过“页面单击自动恢复”将操作成本从“两次点击”降到“一次网页点击”，但要实现完全无额外交互，建议继续推进原生主通道能力验证与落地。

补充结论：
- 要彻底消除“点击播放无声/需点网页激活”，应将发声核心从 WebView 迁移至 ArkTS 原生 TTS 主线路。
- 在当前工程阶段，应采用“原生主线路 + 自动回退链路”策略：既保证体验上限，也保证发布稳定性。

## 10. 技术线路与整体构造思路（详细）
### 10.1 设计目标
- 目标 1：朗读必须可用，优先保证“有声”而不是“单一路径纯净”。
- 目标 2：支持阅读器模式与网页模式共用一套朗读控制。
- 目标 3：在复杂内核限制下可观测、可诊断、可回退。

### 10.2 总体分层
- 表现层（ArkUI）
  - 控制条交互：播放、暂停、上一句、下一句、停止。
  - 状态反馈：当前进度、模式、错误信息。
- 应用编排层（Index.ets）
  - 会话管理：构建句子、切换模式、调度连播、处理用户控制。
  - 策略管理：固定原生优先，失败自动回退 Web 链路。
  - 并发控制：代次锁与命令锁。
- 能力层（Web 与 Native）
  - Web 能力：页面文本抽取、DOM 高亮、speech/audio 回退。
  - Native 能力：AVPlayer 播放云端音频，承担原生主通道与兜底。

### 10.3 端到端技术线路
- 线路 A：启动与文本准备
  - 设置弹窗触发 ttsRequestToken。
  - Index 监听 onTtsRequest。
  - 根据模式从 readerParagraphs 或页面可见文本构建 ttsUnits。
- 线路 B：播放策略选择
  - 默认主线路：优先原生播放器。
  - 自动回退线路：原生失败后切换 Web 语音链路。
  - 任一路径失败后继续回退，不中断用户会话。
- 线路 C：播放调度
  - native 模式：基于播放器 state=completed 推进下一句。
  - audio 模式：基于 Web onended 与探针状态推进。
  - speech 模式：按时长估算推进。
- 线路 D：交互控制
  - 暂停/恢复通过统一入口更新锁与代次。
  - 上一句/下一句先切索引，再按当前模式触发对应播放。

### 10.4 为什么采用双轨策略而非一次性切换
- 原因 1：设备与 SDK 差异导致原生能力存在不可用样本。
- 原因 2：WebView 与网络源质量在不同机型上波动明显。
- 原因 3：线上稳定性需要“自动回退链路”保障快速止损。
- 结论：主线路 + 回退链路是当前阶段最符合工程现实的方案。

## 11. 开发核心难点与解法复盘
### 11.1 难点一：WebView 自动播放限制导致首击无声
- 现象：ArkUI 点击播放后无声，点击网页后恢复。
- 根因：User Activation 约束，ArkUI 点击不等价网页手势激活。
- 解法：
  - 增加 tap-resume 注入，网页一次点击自动续播。
  - 保留原生与 Web 的回退路径，保证至少一条链路可发声。

### 11.2 难点二：原生播放器状态机错误（prepare 时机）
- 现象：BusinessError，提示当前状态不支持 prepare。
- 根因：设置 url/fdSrc 后立即 prepare，未等待 initialized/stopped。
- 解法：
  - 引入 waitNativePlayerReady。
  - 仅在可用状态进入 prepare/play。
  - 增加状态日志字段定位时序异常。

### 11.3 难点三：连播打断与暂停后仍触发下一句
- 现象：前句未播完被后句打断；暂停后仍跳下一句。
- 根因：native 模式曾使用估算时长推进，与真实播放完成不同步。
- 解法：
  - native 模式改为 state=completed 驱动切句。
  - 代次锁与播放标记双重校验，暂停后不再推进。

### 11.4 难点四：多分支重复逻辑导致行为不一致
- 现象：不同入口（resume/next/auto-next）失败处理不一致。
- 根因：重复代码过多，维护时容易分支漂移。
- 解法：
  - 抽取统一失败处理方法。
  - 收敛 resume 主流程，降低分支复杂度。

### 11.5 难点五：云端音源质量不稳定
- 现象：部分音源返回 JSON/500/超时，无法稳定用于 native。
- 根因：第三方音源策略、反爬限制、格式与可达性不一致。
- 解法：
  - 多音源轮询与 direct/cache 双尝试。
  - 对 responseCode、content length、错误信息进行结构化记录。
  - 将“源可用性”与“播放器时序问题”分离诊断。

### 11.6 难点六：调试与体验的平衡
- 现象：排障期需要大量日志，发布态又需简洁界面。
- 解法：
  - 调试阶段放大日志面板，快速定位。
  - 稳定后简化 UI 与日志逻辑，保留核心可观测字段。

## 12. 当前版本的工程取舍
- 保留：
  - 原生优先策略、自动回退链路、核心连播与控制能力。
- 收敛：
  - 模式切换入口、非必要调试 UI、冗余分支与重复失败处理逻辑。
- 后续可继续演进：
  - 在条件允许下替换 AVPlayer 为系统级 @ohos.tts 主通道。
  - 用事件驱动替代轮询驱动，进一步降低时序竞态。
